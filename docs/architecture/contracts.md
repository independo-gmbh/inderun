# IndeRun Data Contracts

This document describes the structured data exchanged between the application and the IndeRun engine. These contracts ensure type-safe, platform-agnostic communication across Web, iOS, and Android environments.

## `TaskRequest`

The `TaskRequest` object represents a single unit of work to be executed by the engine. It is the input payload for the `run()` method.

### Key Fields

* **`constraints`**: Defines the constraints used by the router to select a provider. This determines whether the request targets local resources (`on_device`) or remote services (`cloud`).
* **`task`**: A descriptor that defines the nature of the work (e.t. `text_to_text`). This is used by providers to match their capabilities against the incoming request.
* **`prompt` / `messages`**: The actual content of the task. Use `prompt` for simple single-turn requests, or `messages` for conversation-style interactions.
* **s `requestId`**: An optional, caller-provided string used for idempotency and correlating logs across systems.
* **`authContextRef`**: A secure reference to credentials stored in the host's secure storage. Raw secrets should never be placed directly in the request payload.

## `TaskResult`

The `TaskResult` object is the enriched output returned after a successful execution attempt. It contains the generated content and important runtime telemetry.

### Key Fields

* **`output`**: The result of the task (e.g., the generated text).
* **`runId`**: A unique identifier generated by the engine to track this specific execution attempt throughout its lifecycle.
* **`telemetry`**: Detailed runtime metadata that can be used for monitoring and observability:
    * `providerUsed`: The identity of the provider that handled the request.
    * `totalMs`: The total duration of the execution in milliseconds, useful for measuring latency and optimizing routing policies.

## Error Taxonomy (`IndeRunException`)

When an operation fails, the engine throws a standardized `IndeRunException`. This exception is categorized by an `errorClass` to allow developers to implement intelligent fallback behaviors in their user interfaces.

| Error Class | Meaning & Recommended UI Behavior |
| :--- | :--- |
| `CapabilityMismatch` | The selected provider cannot fulfill the request (e.g., missing a required capability). Consider retrying with a different mode/constraints. |
| `Offline` | The device has no network connectivity. Switch to `on_device` execution or prompt the user to reconnect. |
| `Unavailable` | The requested provider or endpoint is currently unavailable. Check configuration or try again later. |
| `AuthError` | Authentication failed or credentials could not be retrieved. Verify your secure storage/auth context. |
| `RateLimited` | The execution exceeded the allowed quota for the provider. Implement exponential backoff. |
| `Timeout` | The operation did not complete within the allotted time. |
| `Internal` | An unexpected error occurred within the engine or a provider. |
